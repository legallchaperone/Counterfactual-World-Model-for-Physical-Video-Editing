## Planner Schema

The only valid planner I/O schema for new SFT train/eval data is
`e2w.planner_io.v6_executable.v1`.

It supports both `remove` and `add` through `task_type`, `edit_prompt`, and
`quadmask_spec.operation`. This add/remove support is the operation axis. It is
separate from the grounding axis, which requires executable geometry:

- `quadmask_spec.primary.keyframes[].bbox_xyxy_norm1000`
- `quadmask_spec.primary.keyframes[].positive_points_norm1000`
- `quadmask_spec.affected.grid_shape`
- `quadmask_spec.affected.frame_ranges[].cells`

The old prompt schema:

```json
{"quadmask_spec": {"primary": {}, "affected": {}, "keep": {}}}
```

is archived-only. It must not be used for new training, eval, smoke, or
canonical forward-pass input. Planner output parsing is strict: a complete
top-level JSON object with planner keys is required; nested target-object JSON
fragments are parse failures.

## Planner Prompt

The canonical SFT planner prompt is the final-rule-only prompt generated by
`tools/e2w_v0_common.py::build_planner_user_prompt`.

Prompt layout:

1. planner role and `e2w.planner_io.v6_executable.v1`
2. task operation
3. full schema JSON
4. current `video_id` and user request
5. final rules

Do not embed one-shot wrapper examples in train/eval prompts. Wrapper examples
such as `input_video_id`, `input_user_request`, `good_complete_output`, or
`output_excerpt` caused the current LoRA to copy wrapper keys instead of
returning the required top-level planner JSON.

For remove tasks, the final rules require
`counterfactual_expectation.if_removed` to be target-free because that text is
copied into the VACE prompt. It must not name the removed target, aliases,
visible target parts, target material words, or negative forms such as
`no <target>`, `without <target>`, `<target> is removed`, or
`<target> is no longer present`.

This is the canonical full smoke forward pass. It has one valid planner path: run the SFT VLM planner checkpoint and save `raw_output.txt` plus `raw.pred.json` for every selected sample.

Run from `/home/cwx/E2W` with the project env Python. Pick a free GPU first:

Then run the whole v0.2 chain:

```bash
cd /home/cwx/E2W
RUN_NAME=e2w_v0_2_forward_$(date -u +%Y%m%dT%H%M%SZ)
/data/cwx/conda/envs/edit2world-phase1-real/bin/python tools/run_v02_qwen_vace_smoke.py \

  --operation remove \
  --cuda-visible-devices 4 \

```

Replace `4` with the free physical GPU index. In Codex, the default sandbox may hide CUDA devices; request outside-sandbox execution for the real forward pass if `torch.cuda.is_available()` is false or `/dev/nvidia*` is missing.

The orchestrator regenerates artifacts in this order:

1. `eval_vlm_planner.py`
2. `build_quadmask_from_spec.py`
3. `run_first_frame_edit.py --backend qwen_image_edit`
4. `run_vace_v0.py --run-vace`
5. `package_v02_qwen_vace_smoke.py`
6. `artifact_freshness.json`

A clean run must contain `planner_pred/<sample-id>/raw_output.txt` and `planner_pred/<sample-id>/raw.pred.json` for every sample. If those files are missing, the run did not use the SFT VLM planner and is not a valid forward pass.

## v0.3 Quadmask VACE

v0.3 upgrades the VACE stage contract. It does not claim trained visual control yet.

Canonical v0.3 direct experiment wrapper:

```bash
tools/run_vace_v03_quad_experiment.py
```

That wrapper calls:

```bash
tools/run_wan_vace_quad_i2v.py
```

The quad runner expands the Wan-VACE control/context branch from the legacy 96-channel input to a 416-channel E2W context and records:

- `quadmask_passed_to_backend_command`
- `quadmask_consumed_by_backend`
- `quadmask_backend: e2w_quad_i2v`
- `quadmask_alignment`
- `generation_mask_mode`

The binary generation mask only says where VACE may generate. The semantic edit mask remains `quadmask.npy` with values `0/63/127/255`.

Requested add-mug experiment template:

```bash
cd /home/cwx/E2W
RUN_DIR=/data/cwx/E2W/runs/e2w_v0_3_quad_vace_add_0076_$(date -u +%Y%m%dT%H%M%SZ)
/data/cwx/conda/envs/edit2world-phase1-real/bin/python tools/run_vace_v03_quad_experiment.py \
  --src-video /home/cwx/E2W/runs/e2w_v0_2_full_cuda_20260602T0720Z/0076/edited_video.mp4 \
  --prompt "Add a yellow mug on the rotating turntable in front of the spotlight" \
  --quadmask-npy /home/cwx/E2W/runs/e2w_v0_2_full_cuda_20260602T0720Z/0076/quadmask.npy \
  --operation add \
  --run-dir "$RUN_DIR" \
  --generation-mask-mode quadmask-editable \
  --align-quadmask nearest \
  --cuda-visible-devices 4 \
  --sample-steps 8 \
  --run-vace
```

For `0076`, explicit `--align-quadmask nearest` is required because the v0.2 quadmask has 80 frames and the input VACE video has 81 frames.

## Contract Notes

- Do not weaken or delete the schema/prompt/parser checks in
  `tests/test_v02_contracts.py` to make a run pass. Fix the data, prompt, or
  planner instead.
- Planner train/eval user prompts must use `e2w.planner_io.v6_executable.v1`;
  use `tools/rewrite_planner_user_prompt_schema.py` to archive and rewrite
  existing JSONL prompts.
- Qwen first-frame prompts name the actual target label, aliases, and visual descriptor. They must not use `primary subject`.
- Qwen prompts protect non-target objects and only ask to fill target pixels with plausible local background.
- Qwen-Image-Edit currently consumes image + text prompt only. `first_frame_edit_metadata.json` records `target_mask_consumed_by_backend: false`.
- VACE prompts are neutral continuity prompts. They must not contain target aliases or removal wording such as `remove`, `delete`, `erase`, or `removed subject`.
- VACE prompts must not mention the removed target or visible target subparts/materials, even as negative wording such as `no <target>`.
- If candidate planner text for a VACE prompt mentions the removed target or target subparts, the planner/export stage must fail with a contract error. Do not silently drop the sentence or use a neutral fallback.
- If no target-free semantic line remains for the VACE prompt, fail instead of using generic neutral text.
- When a visible non-target object count is clear and relevant, preserve it explicitly in planner/VACE text, for example `one potato`.
- v0 records quadmask/frame-count mismatches as warnings. Do not treat those warnings as visual success or failure unless the stage contract changes.
- v0.3 must record quadmask frame/shape alignment explicitly. Silent frame mismatch is not allowed.
- v0.3 can report `quadmask_passed_to_backend_command: true` only when the command contains `--quadmask_npy`, `--operation`, and `--generation_mask`.
- v0.3 can report `quadmask_consumed_by_backend: true` only when that command actually ran successfully and produced `edited_video.mp4`.
